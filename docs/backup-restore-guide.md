# 备份、恢复与导入导出指南

本文说明 StarNav 的数据导入导出、手动备份、定时备份和恢复策略。

## 1. 数据范围

StarNav 的主要数据分布在 D1 和 KV：

- D1：书签、待审核书签、分类、标签、搜索词统计、操作日志等结构化数据。
- KV：管理员会话、API Token 元数据、WebHook 配置、备份快照、部分系统状态等键值数据。

备份功能会尽量保存当前系统可恢复的核心业务数据。

## 2. 导入导出能力

后台提供：

| 功能 | 说明 |
|---|---|
| 导入预览 | 正式导入前统计有效、无效、重复、缺失分类等信息 |
| 合并导入 | 按归一化 URL 跳过重复书签，保留现有数据 |
| 覆盖恢复 | 清空当前书签、分类和标签后按导入文件重建 |
| 新版导出 | 导出当前新版结构，包含书签、分类和标签 |
| 旧版导出 | 导出兼容旧项目的 `config.json` |
| HTML 导出 | 导出 Netscape Bookmark HTML，可导入浏览器 |
| CSV 导出 | 导出表格格式，便于人工审查或二次处理 |

## 3. 导入建议流程

1. 进入后台。
2. 先创建一份手动备份。
3. 选择导入文件。
4. 查看导入预览。
5. 确认重复项、无效项和缺失分类。
6. 小规模数据优先选择合并导入。
7. 确认要完全替换数据时再使用覆盖恢复。
8. 导入后检查首页、分类、标签和搜索功能。

## 4. 旧版 config.json 迁移

从旧版项目迁移时：

1. 在旧项目后台导出 `config.json`。
2. 部署当前版本。
3. 初始化 D1 和 KV。
4. 登录 `/admin`。
5. 导入旧版 `config.json`。
6. 检查分类、标签和私人书签。
7. 如需层级分类，在分类管理中设置父分类和排序。

兼容策略：

- 旧字段会自动转换为新字段。
- 旧 `catelog` 会自动补齐到分类表。
- 缺失排序值时使用默认排序。
- 标签、可见性等新字段会使用兼容默认值。

## 5. 手动备份

后台可立即创建手动备份。

建议在以下操作前创建手动备份：

- 大批量导入。
- 覆盖恢复。
- 大规模删除或批量修改。
- 升级项目版本。
- 修改 WebHook、Token、系统设置等关键配置。

## 6. 定时备份

项目已内置 Worker `scheduled` 入口，可在 Cron 触发时执行健康巡检和自动备份。

默认仓库没有启用 `wrangler.toml` 的 `[triggers]`，原因是 Cloudflare Cron Trigger 有账号配额限制。确认账号配额后，可自行添加：

```toml
[triggers]
crons = ["0 3 * * *"]
```

也可以在 Cloudflare 控制台的 Worker 设置中手动添加 Cron Trigger。

## 7. 备份保留策略

默认保留最近 30 份备份。超过数量后会清理最旧的记录。

建议：

- 重要升级前单独下载备份文件。
- 长期运营站点可定期将备份文件保存到本地或对象存储。
- 删除备份前确认当前数据和最近一次恢复点可用。

## 8. 恢复策略

恢复方式：

| 方式 | 说明 |
|---|---|
| 覆盖恢复 | 用备份内容替换当前数据，适合回滚 |
| 合并恢复 | 将备份内容合并到当前数据，尽量跳过重复项 |
| 恢复前快照 | 执行恢复前自动创建 `pre-restore` 快照 |

恢复建议：

1. 先下载当前备份。
2. 确认目标备份的创建时间和内容。
3. 优先在测试环境验证。
4. 恢复后立即访问首页和后台检查数据。
5. 如恢复结果不符合预期，可使用恢复前快照回滚。

## 9. 风险提醒

- 覆盖恢复会替换当前数据，执行前务必确认。
- WebHook 可能在恢复或导入后触发相关事件。
- 大量数据导入可能受 Cloudflare Workers 执行时间限制影响。
- 生产环境升级前建议同时保留 D1 导出和后台备份。
