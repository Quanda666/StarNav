# StarNav 第二阶段开发规划

> 阶段定位：第一阶段已基本完成“可用、好用、能扩展”的核心功能闭环。第二阶段重点转向稳定性、可维护性、性能、生态能力、用户体验打磨和长期演进。

---

## 1. 当前项目整体审查结论

### 1.1 第一阶段已形成的核心能力

当前项目已经不再只是一个基础书签导航，而是具备完整管理闭环的 Cloudflare Workers 应用：

- 前台导航
  - 分类树、标签、搜索、视图模式、主题预设、暗黑模式。
  - 收藏、最近访问、快捷键、AI 助理、提交新站。
- 后台管理
  - 书签增删改查、批量编辑、导入导出、健康检测。
  - 分类管理、标签管理、标签合并、AI 标签推荐。
  - 审核队列、访问分析、搜索分析、操作日志。
- 数据与安全
  - D1 存储书签、分类、标签、设置、搜索统计、操作日志。
  - KV 存储管理员认证、私密访问、备份、Token、WebHook 配置。
  - 私密书签、管理员专属书签、不列出书签、访问密码有效期。
- AI 能力
  - 本站强约束回答。
  - 搜索召回增强。
  - 结果卡片化。
  - 分类 / 标签推荐。
  - 标签合并建议。
- 开放生态
  - 公开只读 API。
  - Bearer Token 写入能力。
  - API Discovery。
  - OpenAPI。
  - WebHook。
  - Manifest V3 浏览器插件。
- 工程质量
  - `npm run check` 语法检查。
  - `npm test` Node.js 内置测试。
  - GitHub Actions 质量检查。
  - 部署检查清单、API 文档、插件文档、WebHook 文档、网页版部署教程。

### 1.2 当前主要技术特点

项目采用：

- Cloudflare Workers 模块化入口：`src/index.js`
- API 聚合路由：`src/handlers/api.js`
- 服务层：
  - `siteService.js`
  - `categoryService.js`
  - `tagService.js`
  - `aiService.js`
  - `backupService.js`
  - `webhookService.js`
  - `operationLogService.js`
  - `privateBookmarkService.js`
  - `systemSettingsService.js`
- 页面层：
  - `src/pages/home.js`
  - `src/pages/adminAssets.js`
- 插件：
  - `extensions/browser-bookmark/`
- 测试：
  - `tests/siteService.test.js`
  - `tests/apiErrors.test.js`

### 1.3 当前主要风险和瓶颈

#### A. 单文件职责逐渐变重

以下文件已经承载较多职责：

- `src/handlers/api.js`
  - 同时负责鉴权、路由、导入导出、API Discovery、OpenAPI、多个业务接口。
- `src/pages/home.js`
  - 同时负责 SSR、主题、分类、搜索、私密入口、前台弹窗、前台脚本。
- `src/pages/adminAssets.js`
  - 后台样式、脚本、交互逻辑高度集中。
- `src/services/siteService.js`
  - 覆盖搜索、书签 CRUD、健康检测、导入导出、审核、分析等多个领域。
- `src/services/aiService.js`
  - 覆盖聊天、关键词抽取、RAG、本地回答、标签推荐、分类推荐、标签合并建议。

第二阶段应优先降低这些文件的维护压力。

#### B. 测试覆盖仍偏核心路径

当前测试已经覆盖：

- 搜索排序。
- 权限可见性。
- 导入预览。
- 重复 URL 归一化。
- API 错误结构。

但以下关键能力仍缺少系统测试：

- Bearer Token 权限边界。
- WebHook 签名和事件匹配。
- 备份恢复。
- 私密访问 TTL。
- 分类树循环防护。
- 标签合并与批量应用。
- AI 规则化关键词抽取。
- API Discovery / OpenAPI 输出稳定性。
- 浏览器插件核心请求逻辑。

#### C. 可观测性仍偏基础

已有操作日志和 WebHook，但还缺少：

- 管理后台系统健康概览。
- 最近 Cron 执行记录。
- 最近备份状态。
- 最近 WebHook 投递状态汇总。
- D1 / KV 配置检查。
- API Token 最近使用统计。
- 错误事件的统一归档。

#### D. 文档已经较全，但缺少阶段路线图

现有文档以“功能使用说明”和“部署说明”为主，适合用户使用；第二阶段需要一份面向后续开发的路线图，用于决定接下来做什么、不做什么、按什么顺序做。

## 2. 第二阶段总体目标

第二阶段不建议继续盲目堆功能，而应以“长期可维护的产品化版本”为目标。

### 2.1 核心目标

1. **稳定**
   - 关键功能有测试。
   - 数据操作更安全。
   - 失败场景可追踪。
2. **清晰**
   - API、服务、前端脚本逐步拆分。
   - 新增功能不再继续压到巨型文件中。
3. **高效**
   - 前台加载更快。
   - 后台维护大量书签更省时。
   - 搜索和 AI 响应更可控。
4. **可扩展**
   - 插件、API、WebHook、Token 形成更清晰的第三方接入体系。
   - 后续可接入更多客户端和自动化工具。
5. **易部署**
   - Cloudflare 网页部署路径继续降低门槛。
   - Wrangler 路径继续服务开发者。
   - 初始化、升级、备份、回滚更清晰。

## 4. 迭代一：架构拆分与可维护性提升

### 4.1 目标

降低核心文件复杂度，减少后续修改时的回归风险。

### 4.2 建议拆分方向

#### API 路由拆分

当前：

```text
src/handlers/api.js
```

建议逐步拆为：

```text
src/handlers/api/
  index.js
  auth.js
  sites.js
  categories.js
  tags.js
  ai.js
  analytics.js
  backups.js
  webhooks.js
  tokens.js
  discovery.js
  errors.js
```

第一步可以先拆：

- `errors.js`
- `discovery.js`
- `sites.js`
- `tags.js`

避免一次性大改。

#### 服务层领域拆分

当前 `siteService.js` 职责较多，建议逐步拆为：

```text
src/services/sites/
  visibility.js
  search.js
  health.js
  importExport.js
  analytics.js
  pending.js
  duplicates.js
```

短期不必一次性移动所有代码，可以先把纯函数和测试较多的部分拆出：

- URL 归一化。
- 搜索语法解析。
- 搜索评分。
- 权限可见性判断。
- 导入预览。

#### 前台页面拆分

当前：

```text
src/pages/home.js
```

建议逐步拆为：

```text
src/pages/home/
  render.js
  categories.js
  siteCard.js
  privateAccess.js
  announcement.js
  scripts.js
  styles.js
```

#### 后台资源拆分

当前：

```text
src/pages/adminAssets.js
```

建议逐步拆为：

```text
src/pages/admin/
  styles.js
  scripts/
    core.js
    sites.js
    categories.js
    tags.js
    analytics.js
    backups.js
    webhooks.js
    tokens.js
```

### 4.3 验收标准

- 拆分后 `npm run quality` 通过。
- API 响应结构保持兼容。
- 公开接口路径不变化。
- 每次拆分只移动一类职责，避免大爆炸式重构。
- 为拆出的纯函数补充测试。

## 5. 迭代二：系统健康中心

### 5.1 目标

让管理员一眼知道系统是否正常。

### 5.2 建议功能

新增后台“系统健康”页签，展示：

- D1 连接状态。
- KV 连接状态。
- 数据表完整性。
- 最近一次自动迁移结果。
- 最近一次健康巡检时间。
- 最近一次定时备份时间。
- 最近一次 WebHook 发送结果。
- API Token 数量、最近使用时间。
- 当前公开提交是否启用。
- AI 配置是否完整。
- Cron Trigger 配置提示。
- 当前版本 / 构建时间，可选。

### 5.3 建议接口

```http
GET /api/system/health
```

仅允许后台 cookie 管理员访问。

返回示例：

```json
{
  "code": 0,
  "status": "ok",
  "checks": [
    {
      "name": "D1",
      "status": "ok",
      "message": "NAV_DB 可用"
    },
    {
      "name": "KV",
      "status": "ok",
      "message": "NAV_AUTH 可用"
    }
  ]
}
```

### 5.4 验收标准

- 后台可展示系统健康状态。
- D1 / KV 异常时有明确提示。
- 不泄露敏感信息。
- 检查失败不导致后台整体崩溃。

## 6. 迭代三：测试体系扩展

### 6.1 目标

从“能检查语法和少量核心函数”升级为“覆盖主要业务边界”。

### 6.2 优先补充测试

#### 权限与 Token

- 无 Token 写入返回 401。
- read token 不能写入。
- write token 可以创建书签。
- revoked token 不能使用。
- admin scope 不等于后台 cookie 管理员权限。
- Token 不能管理 Token 自身。

#### WebHook

- `*` 匹配所有事件。
- `site.*` 匹配站点事件。
- `site.create` 精确匹配。
- HTTP URL 被拒绝。
- HMAC 签名格式正确。
- 发送失败时记录最后错误。

#### 备份恢复

- 手动备份生成元数据。
- 合并恢复按 URL 去重。
- 覆盖恢复会创建恢复前快照。
- 删除备份只删除对应备份。

#### 搜索与 AI 规则

- `tag:`、`cat:`、`url:`、`is:` 组合搜索。
- 中文首字母召回。
- 包含关键词严格过滤。
- AI 本地强约束回答不编造。
- 前端传入上下文 ID 时后端重新读取并过滤权限。

#### 分类与标签

- 分类不能移动到自身子孙下。
- 标签合并迁移关联。
- 批量应用标签追加 / 替换模式正确。

### 6.3 验收标准

- `npm run quality` 测试数量明显增加。
- 每次新功能必须至少补充一类测试。
- CI 持续通过。

## 7. 迭代四：数据安全、备份与升级增强

### 7.1 目标

减少误操作和升级失败风险。

### 7.2 建议功能

  - 书签数。
  - 分类数。
  - 标签数。
  - 创建原因。
  - 大小。
  - 备份版本。
- 恢复预览：
  - 将新增多少。
  - 将覆盖多少。
  - 将跳过多少。
  - 可能删除多少。

  - JSON 结构校验。
  - 必要字段校验。
  - URL 格式校验。

- 迁移版本号记录。
- 最近迁移时间记录。
- 迁移失败日志。
- 后台显示当前 schema 状态。

可新增表：

```sql
CREATE TABLE IF NOT EXISTS schema_migrations (
  version TEXT PRIMARY KEY,
  applied_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
  note TEXT
);
```

### 7.3 验收标准

- 恢复前可预览影响。
- 迁移可追踪。
- 备份失败有可见提示。

## 10. 迭代七：AI 助理第二阶段

### 10.1 目标

从“能回答本站书签问题”升级为“能辅助管理和发现问题”。

### 10.2 建议功能

#### AI 管理助手

仅后台管理员可用：

- “帮我找出可能重复的书签”
- “帮我找出没有标签的书签”
- “帮我找出可能分类错误的书签”
- “帮我总结最近无结果搜索词应该补哪些书签”
- “帮我生成标签整理建议”

#### AI 搜索解释

回答中说明：

- 使用了哪些关键词。
- 命中了哪些字段。
- 为什么推荐这些书签。

#### AI 批量建议队列

- 建议分类。
- 建议标签。
- 建议合并标签。
- 建议隐藏失效链接。
- 管理员逐项确认应用。

### 10.3 边界原则

- AI 只能建议，不自动执行高风险写操作。
- 批量替换、删除、隐藏必须二次确认。
- AI 输出不得绕过权限过滤。
- AI 不应看到未授权私密数据。

## 13. 迭代十：性能优化

### 13.1 目标

降低 Worker 响应时间和前端首屏压力。

### 13.2 建议方向

#### 后端

- 搜索 SQL 候选集优化。
- 大量书签时分页策略优化。
- 分类 / 标签元数据缓存。
- 系统设置读取缓存。
- Favicon 请求超时和并发控制。
- 健康检测限流。

#### 前端

- 减少 SSR HTML 中内嵌大脚本体积。
- 管理后台脚本拆包或按需加载。
- 搜索结果 DOM 更新节流。
- 大量卡片时虚拟列表，可选。
- 图片懒加载和尺寸固定，减少布局抖动。

### 13.3 验收指标

## 19. 后续维护规则建议

第二阶段开始后，建议遵循：

1. 新功能必须同步更新文档。
2. 涉及 API 的改动必须考虑 OpenAPI。
3. 涉及写操作的改动必须考虑操作日志和 WebHook。
4. 涉及数据结构的改动必须同步：
   - `schema.sql`
   - `migrationService.js`
   - 部署 / 升级文档
5. 涉及权限的改动必须补测试。
6. 涉及前台私密数据的改动必须检查是否会泄露到 HTML、JS、API 或 PWA 缓存。
7. 大文件拆分优先小步迁移，不做一次性大重构。
